傳輸層的兩大協定，代表可靠性與速度的根本取捨。系統設計面試的預設選擇是 TCP，除非有明確理由才選 UDP。

## 快速對照

| 特性 | UDP | TCP |
|------|-----|-----|
| 連線方式 | 無連線 | 面向連線（三次握手） |
| 可靠性 | 盡力而為交付 | 保證交付 |
| 順序 | 不保證 | 維護 |
| 流量控制 | 無 | 有 |
| 壅塞控制 | 無 | 有 |
| 標頭大小 | **8 bytes** | **20-60 bytes** |
| 速度 | 較快 | 因額外開銷而較慢 |
| 使用場景 | 串流、遊戲、VoIP | 幾乎其他所有情況 |

## TCP：可靠但有額外開銷

TCP（Transmission Control Protocol）提供可靠的、有序的、有錯誤檢查的資料交付。連線是「串流（stream）」——有狀態的長連線，同一串流中的兩則訊息以相同順序到達。

**關鍵特性：**
- 面向連線：資料傳輸前先建立連線（三次握手）
- 可靠交付：保證資料有序且無錯誤地到達
- 流量控制：防止傳送方超過接收方的處理能力
- 壅塞控制：適應網路壅塞，防止崩潰

**適合場景**：幾乎所有 UDP 不適合的場景——HTTP、API、檔案傳輸。

## UDP：快速但不可靠

UDP（User Datagram Protocol）幾乎不在 IP 之上加任何功能，向前衝、聽天命。

**適合場景：**
- 低延遲至關重要（即時應用、遊戲）
- 一些資料遺失可以接受（媒體串流、VoIP）
- 高流量的遙測或日誌（偶爾遺失可接受）
- DNS 查找

> **注意**：除了 WebRTC 之外，瀏覽器對 UDP 的支援並不廣泛。如果設計需要 UDP 的 Web 方案，要考慮瀏覽器用戶只能用 HTTP 串流替代。

## QUIC

較新的協定，目標是提供 TCP 的部分優勢並加入現代化的效能改進（HTTP/3 使用）。可以把它視為更好版本的 TCP，但尚未廣泛普及。面試中通常不需要深入討論。

## 面試提示

- **預設選 TCP**，面試官通常假設如此，不需特別說明
- 若能為 UDP 提出合理論據（低延遲需求且可接受資料遺失），可以額外加分
- POST 不是冪等的（見 [[冪等性]]）
