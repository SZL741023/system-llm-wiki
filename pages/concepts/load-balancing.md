水平擴展（多台伺服器）需要負載平衡來決定每個請求要發送到哪台伺服器。負載平衡分為客戶端負載平衡和專用負載平衡器兩種方式。

## 客戶端負載平衡 vs 專用負載平衡器

**決策樹：**
```
客戶端數量多？
  └─ 否 → 客戶端負載平衡
  └─ 是 → 能容忍更新延遲？
              └─ 是 → 客戶端負載平衡
              └─ 否 → 專用負載平衡器
```

### 客戶端負載平衡

客戶端自己決定要跟哪台伺服器溝通，通常透過查詢 service registry 取得可用伺服器列表，然後直接對其中一台發出請求。

**優點**：非常快且高效，客戶端自己做決定，不增加任何額外延遲。

**適用場景（少量可控客戶端或可接受慢速更新）：**
- **Redis Cluster**：節點之間維護 gossip 協定，客戶端連到任一節點，根據 key 決定發到哪個分片
- **DNS**：回傳輪換過的 IP 位址列表；也是避免負載平衡器單點故障的方式（兩個 LB 用 DNS 輪換）
- **gRPC**：內建客戶端負載平衡，適合內部微服務

### 專用負載平衡器

位於客戶端和後端伺服器之間的伺服器或硬體裝置，負責決定把請求發到哪台伺服器。

每個請求需要額外一跳（先到 LB 再到實際服務），換來的是對伺服器列表的快速更新和對路由方式的細粒度控制。

## L4 vs L7 負載平衡器

### L4 負載平衡器（傳輸層）

在 TCP/UDP 層運作，根據 IP 位址和 port 做路由，**不查看封包的實際內容**。

**關鍵特性：**
- 維護客戶端和伺服器之間的**持久 TCP 連線**（同一 TCP session 的所有後續請求都到同一台伺服器）
- 速度快且高效（封包檢查最少）
- 無法根據應用層資料做路由決策

**什麼時候用**：WebSocket 連線和其他需要持久連線的協定。如果面試中使用 WebSocket，你可能需要 L4 LB。

### L7 負載平衡器（應用層）

在應用層運作，理解 HTTP 等協定，可以**檢查每個請求的實際內容**做出更智慧的路由決策。

**關鍵特性：**
- 終止傳入的連線，並向後端伺服器建立新的連線
- 可以根據請求內容（URL、headers、cookies）路由
- 由於封包檢查，CPU 消耗更多
- 提供更多靈活性和功能（類似 API Gateway 的功能）

**什麼時候用**：基於 HTTP 的流量（涵蓋了除 WebSocket 之外我們討論過的所有協定）。

## 健康檢查與容錯

負載平衡器除了分配流量，也負責監控後端伺服器的健康狀況。

- **TCP 健康檢查**：簡單且高效，檢查伺服器是否接受新連線
- **L7 健康檢查**：向伺服器發出 HTTP 請求，確認回應是成功的（例如 200 狀態碼）

這種自動故障切換能力是負載平衡器對高可用性至關重要的原因——在沒有人工介入的情況下偵測故障並繞道而行。

## 負載平衡演算法

| 演算法 | 說明 | 適用場景 |
|--------|------|---------|
| 輪詢（Round Robin） | 請求依序分配到各伺服器 | 無狀態應用 |
| 隨機（Random） | 請求隨機分配 | 無狀態應用 |
| 最少連線（Least Connections） | 請求發到當前活躍連線最少的伺服器 | **SSE、WebSocket 長連線** |
| 最快回應（Least Response Time） | 請求發到回應時間最快的伺服器 | — |
| IP 雜湊（IP Hash） | 由客戶端 IP 決定哪台伺服器接收請求 | 工作階段持久性 |

> 對需要持久連線的服務（SSE、WebSocket），使用**最少連線**，因為它避免了單台伺服器逐漸積累所有活躍連線的情況。

## 實際的實作選擇

- **軟體**：HAProxy、NGINX、Envoy
- **硬體**：F5 Networks BIG-IP（能擴展到每秒幾億個請求）
- **雲端**：AWS ELB/ALB/NLB、Google Cloud Load Balancing、Azure Load Balancer
