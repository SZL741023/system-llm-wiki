Scalability 指的是系統能隨著需求（使用者、資料量、流量）的增加，透過增加資源來維持效能與穩定性的能力。核心指標是 **吞吐量（Throughput）** 每秒能處理的請求數，以及 **延遲（Latency）** 單一請求處理所需時間。

## 垂直擴展（Vertical Scaling）

升級單一伺服器：加 CPU、記憶體、硬碟。

| 項目 | 說明 |
|------|------|
| 做法 | 把一台機器換成更大的機器 |
| 上限 | 有物理上限（單台機器的最大規格） |
| 複雜度 | 低，改動少 |
| 成本 | 初期低、後期貴 |
| 適合 | 短期、快速解法 |

## 水平擴展（Horizontal Scaling）

增加更多伺服器，透過 [[負載平衡|Load Balancer]] 讓多台機器分擔工作。

| 項目 | 說明 |
|------|------|
| 做法 | 增加機器數量 |
| 上限 | 理論上無限 |
| 複雜度 | 高，需要處理分散式資料一致性 |
| 成本 | 彈性（按需增減） |
| 適合 | 長期、大規模成長 |

> **面試考點**：Horizontal scaling 需要處理分散式資料一致性問題（這是水平擴展的真正挑戰，也是 [[分片（Sharding）]] 和 [[複製（Replication）]] 存在的原因）。Vertical scaling 有物理上限。

## 實務策略

一般來說，企業會先透過**垂直擴展**解決短期需求，等流量持續增加時，再轉向**水平擴展**來支援長期成長。

> **面試注意**：不要過早引入水平擴展的複雜度。一台調校良好的單一資料庫，能走的路比大多數工程師預期的還要遠（見 [[系統設計關鍵數字]]）。
